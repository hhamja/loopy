# 최정예 실리콘밸리 팀은 루프 엔지니어링을 어떻게 하는가

> **한 줄 요약:** 뛰어난 팀은 "에이전트에게 매 턴 프롬프트를 던지는 사람"이 아니라, **에이전트를 프롬프트하는 시스템을 설계하는 사람**이 된다. 이들의 진짜 산출물은 코드가 아니라 *루프*다.

---

## 0. 관점의 전환 — 무엇이 진짜 일인가

평범한 팀은 이렇게 일한다:

```
사람 → 프롬프트 → 결과 확인 → 다시 프롬프트 → 결과 확인 → ...
```

사람이 루프의 일부다. 매 반복마다 사람의 주의(attention)가 병목이 된다. 에이전트가 아무리 빨라도, 사람이 결과를 읽고 다음 지시를 내리는 속도 이상으로는 못 간다.

최정예 팀은 이렇게 일한다:

```
사람 → 루프를 설계 → [ 실행 → 피드백 수집 → 자기교정 ]* → 검증 가능한 정지 → 사람이 게이트에서 승인
                       └─────────── 사람 없이 도는 부분 ───────────┘
```

**핵심 통찰:** 루프 엔지니어링이란 *에이전트에게 프롬프트를 던지는 일*이 아니라 *에이전트를 프롬프트하는 시스템을 만드는 일*이다. 루프 = `실행 → 피드백 수집 → 자기교정`을, **기계로 검증 가능한 정지 조건**이 성립할 때까지 반복하는 것.

사람의 자리는 "매 턴"이 아니라 "되돌릴 수 없는 경계(gate)"로 이동한다. 이것이 생산성의 차이를 만든다.

---

## 1. 루프의 해부학

좋은 루프는 네 부분이 명확히 분리되어 있다.

| 구성요소 | 역할 | 나쁜 팀의 실패 방식 |
|---|---|---|
| **Runner (실행자)** | 실제로 코드를 짜고 명령을 돌린다 | 실행자가 자기 결과를 스스로 "좋다"고 판정 |
| **Feedback (피드백)** | 테스트·린트·타입체크·런타임 관찰 등 *객관적 신호* | "돌아가는 것 같다" 같은 주관적 신호에 의존 |
| **Checker (검증자)** | 루브릭 기준으로만 통과/실패를 채점 | 검증자가 없거나, 실행자와 컨텍스트를 공유 |
| **Stop condition (정지 조건)** | 언제 끝나는지를 *기계적으로* 정의 | "될 때까지" 같은 무한 루프 |

이 네 개가 뭉개지면 루프는 그럴듯한 헛것을 만든다. 분리되어 있으면 사람 없이도 신뢰할 수 있게 돈다.

---

## 2. 비타협 원칙 세 가지

최정예 팀이 예외를 두지 않는 세 가지. (이 레포의 loop-engineering 방법론도 정확히 이 세 축 위에 서 있다.)

### 원칙 1 — 만드는 자는 자기 일을 채점하지 않는다 (Maker ≠ Checker)

구현하는 주체(maker)와 채점하는 주체(checker)를 **다른 컨텍스트로 물리적으로 분리**한다.

- Maker는 마음껏 짠다.
- Checker는 **새 컨텍스트, 읽기 전용, 루브릭이 유일한 기준.** maker가 무슨 생각으로 짰는지 모른다. 코드와 루브릭만 본다.
- Checker는 "리포트"를 반환하고, 그 리포트를 상태 파일에 적용하는 것은 또 다른 단계다.

> **왜?** LLM은 자기 산출물에 관대하다. "내가 짠 게 맞다"는 확증편향을 구조적으로 차단하려면, 채점자가 생성자의 의도를 아예 몰라야 한다. Anthropic·OpenAI 내부의 에이전트 평가 파이프라인이 generator와 grader(LLM-as-judge)를 분리하는 이유와 같다.

### 원칙 2 — "완료"는 주장이지 증명이 아니다 (Machine-verifiable stops)

모든 정지 조건은 **기계로 확인 가능**해야 한다:

- 명령이 exit 0으로 끝난다
- 파일이 존재한다
- 출력이 기대값과 일치한다

"코드가 깔끔해졌다", "사용자 경험이 좋아졌다" 같은 **주관적 기준은 루브릭에서 금지**된다. 주관적 기준은 사람의 판단을 매 턴 다시 불러들이고 — 그 순간 루프는 죽는다.

> 성공 기준이 명확하면 에이전트가 독립적으로 반복할 수 있다. 모호한 기준("일단 되게 해줘")은 계속 확인을 요구하게 만든다.

### 원칙 3 — 메모리는 컨텍스트가 아니라 디스크에 있다 (Disk over context)

다음 세션이 필요로 하는 모든 것은 **디스크의 상태 파일**에 있어야 한다.

- 컨텍스트 창은 휘발성이고 유한하다. 여기에 진실을 두면 요약·압축 과정에서 손실된다.
- 상태를 디스크에 두면, **완전히 새로운 세션이 그 디렉터리만 읽고 멈춘 지점부터 이어갈 수 있다.**
- `state.md`가 진입점. `goal.md`(목표), `rubric.md`(합격 기준), `memory.md`(누적 교훈), `review.md`(사람 리뷰 요약)가 뒤를 받친다.

> 이것이 "에이전트 우선 설계(agent-first design)"의 핵심이다: 상태는 컨텍스트가 아니라 디스크에, 완료는 주관이 아니라 기계 검증으로.

## 3. 최정예 팀이 *실제로* 하는 것들

원칙을 실천으로 옮기는 구체적 무브들.

### 3.1 평가가 먼저다 (Eval-driven development)

기능을 짜기 전에 **채점 방법을 먼저 짠다.**

- "검증 추가" → *잘못된 입력에 대한 테스트를 먼저 쓰고, 통과시키기*
- "버그 수정" → *버그를 재현하는 테스트를 먼저 쓰고, 통과시키기*
- "X 리팩터링" → *전후로 테스트가 통과하는지 확인하기*

루브릭(rubric)이 코드보다 먼저 존재한다. 이게 있어야 루프가 스스로 "합격/불합격"을 말할 수 있다. 평가 없는 에이전트 루프는 방향타 없는 배다.

### 3.2 사이클은 "구현 → 채점(단 한 번) → 상태 갱신" 형태다

성숙한 루프의 한 사이클:

```
1. 구현        (maker가 디스크 상태로부터 매번 새로 시작 — 이전 세션을 이어받지 않음)
2. 채점        (checker가 사이클 끝에 딱 한 번 — 파일 편집마다가 아니라 "단계 게이트")
3. 상태 갱신   (루브릭 체크박스 갱신 → state.md 재작성 → 메모리 기록 → review.md 덮어쓰기)
```

**채점은 파일 편집마다가 아니라 사이클 끝에 한 번**(phase gate). 매 편집마다 채점하면 토큰이 폭발하고 신호가 흐려진다.

### 3.3 안전 레일은 항상 켜져 있다 (No unbounded loops)

- `max_iterations` 상한 — 반드시 존재.
- **3회 연속 실패 시 에스컬레이션** — 같은 벽에 계속 머리를 박지 않는다.
- "합격할 때까지 무한 반복"은 **금지**. 이건 원칙이 아니라 규율이다.

> CLAUDE.md의 에이전트 우선 설계 원칙과 동일: "무한 반복을 금지할 것 — 반복 횟수 상한과 정지 조건을 반드시 둘 것."

### 3.4 결정 게이트 — "물어봐도 되나"가 아니라 "되돌릴 수 있나"

행동을 하기 전에 **되돌림 가능성 × 영향도**로 분류한다.

| 분류 | 예시 | 행동 |
|---|---|---|
| **되돌릴 수 있고 국소적** | 파일 편집, 테스트, 로컬 커밋, 작업 브랜치 push | **그냥 한다.** 다시 묻지 않는다. |
| **되돌릴 수 없거나 영향이 큼** | 보호 브랜치 병합, 릴리스, 배포, 외부 전송, 비용 발생, 파괴적 삭제 | **사람 게이트에서 멈춘다.** |

판단 기준은 **"이걸 되돌릴 수 있는가?"** 이지 **"물어봐도 되는가?"** 가 아니다. 그리고 이 분류는 사람의 기분이 아니라 **훅/스크립트로 기계적으로 강제**된다(`decision_gate.sh`).

이게 왜 중요한가: 되돌릴 수 있는 일에까지 사람 승인을 받으면 루프가 느려지고, 되돌릴 수 없는 일을 자동으로 하면 사고가 난다. 최정예 팀은 이 경계를 코드로 못박는다.

### 3.5 값싼 정찰 — 컨텍스트를 아껴라

메인 에이전트의 비싼 컨텍스트를 넓은 코드 읽기에 낭비하지 않는다.

- 사이클 전 코드베이스 정찰(파일 지도, 심볼 위치, 관례 파악)은 **값싼 읽기 전용 에이전트**(예: haiku 기반 explorer)에게 위임.
- 결과만 요약해서 받는다. 파일 덤프를 메인 컨텍스트에 붓지 않는다.

### 3.6 비용 규율 (Cost discipline)

루프·검증자·서브에이전트는 토큰을 태운다. 최정예 팀은 이것을 엔지니어링 대상으로 다룬다.

- 전면 실행 전에 `--verify-only`(채점만) 또는 `--once`(한 사이클만)로 저렴하게 확인.
- **결정론적 작업은 프롬프트가 아니라 스크립트로.** 예산 카운팅, 훅 판정, 상태 파싱, 토큰 추정 — 이런 건 LLM에게 시키지 않는다.
- 무거운 실행자(예: 외부 CLI)의 출력은 **메인 컨텍스트에 들이지 않는다.** 로그 파일로 흘리고, 메인은 마지막 요약만 읽는다.

## 4. 그들이 피하는 안티패턴

| 안티패턴 | 왜 나쁜가 | 대신 |
|---|---|---|
| **자기 채점** | maker가 자기 편이 되어 헛것을 통과시킴 | maker ≠ checker, 새 컨텍스트 검증자 |
| **"될 때까지" 루프** | 무한 반복, 토큰 폭발, 정지 불가 | `max_iterations` + 연속 실패 에스컬레이션 |
| **주관적 완료 기준** | 사람을 매 턴 불러들여 루프가 죽음 | exit 0 / 파일 존재 / 출력 일치 |
| **컨텍스트에 상태 저장** | 압축·요약에서 손실, 세션 재개 불가 | 디스크 상태 파일(`state.md` …) |
| **모든 것에 사람 승인** | 병목, 루프가 사람 속도로 하향 | 되돌림 가능성으로 분류, 게이트만 사람 |
| **매 편집마다 채점** | 신호 잡음화, 비용 폭발 | 사이클 끝 단일 phase-gate 채점 |
| **투기적 유연성** | 한 번 쓸 코드에 추상화, 안 오는 시나리오 방어 | 문제를 푸는 최소 코드만 |

## 5. 성숙도 모델 — 우리 팀은 어디쯤인가

```
레벨 0  사람이 매 턴 프롬프트를 던진다. 루프가 없다.
레벨 1  스크립트로 반복은 하지만 정지 조건이 주관적이다.
레벨 2  기계 검증 가능한 정지 조건이 있다. 하지만 maker가 자기를 채점한다.
레벨 3  maker/checker가 분리되고, 상태가 디스크에 있다. 세션을 재개할 수 있다.
레벨 4  결정 게이트가 기계적으로 강제된다. 사람은 되돌릴 수 없는 경계에만 선다.
레벨 5  비용이 엔지니어링된다. 정찰·검증·실행이 비용 계층에 맞게 분리된다.
```

레벨 3부터가 "루프 엔지니어링"이라 부를 만한 지점이다. 레벨 4–5가 최정예의 영역.

## 6. 실전 체크리스트

새 루프를 설계할 때 스스로에게 던지는 질문:

- [ ] **정지 조건이 기계로 확인 가능한가?** (명령 exit 0 / 파일 존재 / 출력 일치)
- [ ] **채점하는 주체가 만드는 주체와 다른 컨텍스트인가?**
- [ ] **채점자의 유일한 기준이 루브릭인가?** (숨은 주관 기준이 없는가)
- [ ] **루브릭이 코드보다 먼저 존재하는가?** (eval-driven)
- [ ] **새 세션이 디스크만 읽고 이어갈 수 있는가?**
- [ ] **`max_iterations`와 연속 실패 에스컬레이션이 있는가?**
- [ ] **되돌릴 수 없는 행동에만 사람 게이트가 걸려 있는가?**
- [ ] **그 게이트가 기분이 아니라 스크립트로 강제되는가?**
- [ ] **넓은 읽기를 값싼 에이전트에 위임했는가?**
- [ ] **결정론적 작업을 프롬프트가 아니라 스크립트로 옮겼는가?**
- [ ] **전면 실행 전에 `--once` / `--verify-only`로 저렴하게 검증했는가?**

이 11개에 전부 "예"라고 답할 수 있으면, 사람 없이 밤새 도는 루프를 신뢰할 수 있다.

## 7. 이 레포(`loop-harness`)에서의 구현 매핑

이 문서의 원칙들은 추상론이 아니라, 이 프로젝트에 코드로 구현되어 있다.

| 원칙 | 이 레포의 구현 |
|---|---|
| Maker ≠ Checker | `implementer`(Codex CLI 또는 메인 에이전트) vs `verifier` 서브에이전트(읽기 전용, 새 컨텍스트) |
| 기계 검증 정지 | `.claude/loop/rubric.md` — 주관 기준 금지, 체크박스 기반 |
| 디스크 메모리 | `.claude/loop/` — `state.md`·`goal.md`·`rubric.md`·`memory.md`·`review.md` |
| 결정 게이트 | `decision_gate.sh` — 되돌림 가능성으로 분류, 훅으로 강제 |
| 값싼 정찰 | `explorer` 에이전트(haiku, 읽기 전용) |
| 사이클 실행 | `/loop-harness:loop-run` (`--once`, `--verify-only` 지원) |
| 상태 요약 | `/loop-harness:loop-status` |
| 원칙 감사 | `/loop-harness:loop-audit` — 루프가 원칙을 지키는지 스스로 채점 |

> 즉, 이 레포는 "최정예 팀의 루프 엔지니어링"을 **설치 가능한 하네스**로 만든 것이다. 원칙을 읽고 매번 지키려 애쓰는 대신, 원칙을 어기기 어렵게 만든 도구.

---

### 맺음말

루프 엔지니어링의 본질은 화려한 프롬프트가 아니다. **신뢰할 수 있는 정지 조건, 이해관계가 분리된 채점자, 디스크에 사는 상태, 그리고 되돌릴 수 없는 경계에만 서는 사람.** 최정예 팀이 남들보다 빠른 이유는 더 좋은 모델을 써서가 아니라, 사람을 루프 안쪽에서 *바깥쪽 게이트*로 빼내는 시스템을 설계했기 때문이다.

> 좋은 루프의 최종 신호: diff에 불필요한 변경이 줄고, 과한 복잡화로 인한 재작성이 줄며, 실수를 저지른 *후*가 아니라 구현 *전*에 명확화 질문이 먼저 나온다.
