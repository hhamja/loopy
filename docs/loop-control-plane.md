# 루프 엔지니어링

> **핵심 명제:** 루프의 자율 반경 = 검증기의 신뢰도 × 되돌림 가능성.
> 두 항 모두 프롬프트가 아니라 인프라로 만든다.

에이전트를 매 턴 프롬프트하는 일이 아니라, 에이전트를 프롬프트하는 **하네스를 설계하는** 일이다. 검증기가 약하면 루프를 아무리 길게 돌려도 나오는 건 "테스트를 통과하는 쓰레기"다.

정지 조건은 항상 기계 검증 가능이어야 한다 — `exit 0`, 테스트 통과, eval 임계값. 사람은 비가역 경계에만 등장한다.

*2026 중반 기준 통상 업계 지식. 도구·프로젝트 비종속.*

## 1. 제어 평면

```
사람 (Intake: 목표·스펙)
  ↓
L1 오케스트레이터 — 상태 소유. plan → task DAG → dispatch → integrate
  ↓ dispatch
L2 워커 (maker) — 격리 sandbox / worktree. 병렬 시 파일 스코프 무교차
  ↓ diff
L3 체커 (verifier) — maker ≠ checker. 독립 컨텍스트
  ↓ pass/fail + evidence
L4 게이트 — CI. 결정론적 → 비결정적 순. 비가역 명령군 하드블록
  ↓ green + clean
L5 사람 게이트 — 비가역 경계에서만

전 계층 관통: durable state · tracing · 예산 상한 · 서킷브레이커 · 부수효과 원장
```

### 두 절단면: 제어 평면(L1~L5) ↔ 책임(ETCLOVG)

위 L1~L5는 **제어 평면**이다 — "누가 무슨 순서로 도느냐"(런타임 역할). 하네스를 다르게 자르는 축이 **ETCLOVG 책임 택소노미**다 — "하네스가 제공해야 하는 기능이 뭐냐". 둘은 같은 걸 다른 해상도로 자른 게 아니라 **직교하는 절단면**이다. 레이어를 5개에서 7개로 "늘리는" 문제가 아니라 매핑의 문제다.

| ETCLOVG 책임 | 이 문서에서의 위치 |
|---|---|
| **E**xecution | L2 워커 (sandbox / worktree) |
| **T**ooling | 툴콜 — 유일하게 독립 서술이 얇음. 권한·스키마·샌드박싱이 여러 절에 분산 |
| **C**ontext | §2 컨텍스트 격리 + durable state |
| **L**ifecycle | L1 오케스트레이터 plan→dispatch→integrate + 예산 상한 · 서킷브레이커 |
| **O**bservability | 관통 tracing + §6 |
| **V**erification | L3 체커 + L4 게이트 + 홀드아웃 |
| **G**overnance | L5 사람 게이트 + 가역성 T0~T2 + 부수효과 원장 + kill-switch |

즉 7개 책임은 이미 전부 커버된다 — 역할 축으로 팩터링됐을 뿐. 굳이 보강할 곳은 **Tooling** 하나다.

## 2. 오케스트레이터 ↔ 워커

나누는 건 "작업"이 아니라 **컨텍스트와 권한**이다.

**오케스트레이터의 유일한 자산은 깨끗한 컨텍스트다.** 직접 파일을 읽기 시작하는 순간 실패한다 — 워커의 탐색 쓰레기가 역류하면 수십 턴 뒤 계획 능력이 무너진다. 워커는 격리 컨텍스트에서 일하고 **요약본만** 반환한다. orchestrator-worker 패턴의 실제 이유는 병렬성이 아니라 **컨텍스트 격리**다.

| 요소 | 구현 |
|---|---|
| 워커 격리 | `git worktree` 하나당 워커 하나. 컨테이너면 더 좋다 |
| 반환 인터페이스 | diff + 테스트 결과 + 3줄 요약. 대화 로그 반환 금지 |
| 예산 집행 | 워커당 토큰/툴콜/벽시계 상한. 초과 시 오케스트레이터가 킬 |
| 팬아웃 폭 | 읽기 전용 정찰은 넓게(5~10), 쓰기 워커는 좁게(1~3) |
| 컨텍스트 | 태스크마다 fresh 컨텍스트를 durable state에서 재구성 |

**아키텍처 선택:** 자유방임 "에이전트 스웜"보다 제약된 상태기계/그래프(LangGraph, Temporal, 자체 DAG 러너). 그래프가 결정성을 주는 건 아니다 — LLM 호출은 여전히 비결정적이다. 얻는 건 **탐색 공간의 상한**이다.

**비용 사실 (Anthropic, 검증됨):** 리서치 시스템은 orchestrator-worker로 단일 에이전트 대비 내부 eval에서 **90.2% 개선**(Opus 4 리드 + Sonnet 4 서브에이전트). 토큰은 **일반 챗 대비 ~15배**. 단, 단일 에이전트도 챗 대비 ~4배이므로 **단일 에이전트 대비로는 약 4배**다. 성능 분산의 80%가 토큰 사용량으로 설명된다.

> ⚠️ 기준선 주의: "멀티에이전트 = 15배"는 챗 기준이다. 단일 에이전트 기준으로 오독하는 문헌이 많다.

**실무 규칙:** 병렬 쓰기 워커는 파일 스코프가 겹치면 안 된다. 겹치면 머지 컨플릭트 해결에 드는 에이전트 시간이 병렬화 이득을 잡아먹는다. 스코프를 못 나누면 그냥 직렬로 돌려라. 팬아웃 전에 오케스트레이터가 **인터페이스 계약을 먼저 고정**한다.

## 3. Maker / Checker: 분리의 조건

만든 놈이 채점하지 않는다. 체커를 붙이는 건 쉽고, **의미 있게 분리하는 게 어렵다.** 같은 모델에게 "리뷰해줘"라고 하면 sycophantic rubber-stamp가 나온다.

분리가 작동하려면 최소 하나는 달라야 한다:

1. **정보 비대칭** — 체커는 maker가 못 본 홀드아웃 테스트/스펙을 갖는다. 가장 강력하다.
2. **권한 비대칭** — 체커는 쓰기 권한이 없다. 코드도 테스트도 못 고친다.
3. **인센티브 비대칭** — 프롬프트가 "통과시켜라"가 아니라 "이 diff가 스펙을 위반하는 방식을 찾아라"다. **반증 과제로 프레이밍**한다.

**결정적 안티패턴:** maker가 자기 테스트를 쓰고 체커가 그 테스트로 검증. 이건 검증이 아니라 동어반복이다. 테스트 작성 권한과 구현 권한은 다른 에이전트에, 다른 순서로(테스트 먼저 → 잠금 → 구현). **런 중 수용 기준 동결 — 기준 변경은 사람만.**

**체커 모델 티어 — 증명서 유무로 가른다:**

- 검증에 증명서가 있으면 **싼 모델**: 테스트가 이미 실패 지점을 지목했고 체커는 확인만 하면 됨
- 증명서가 없으면 **비싼 모델**: 스펙 위반·아키텍처 판단은 체커가 반례를 스스로 발명해야 함. (SpecBench 실무 권고와 일치 — 복잡한 태스크에선 test-time compute·테스트 추가보다 **가장 강한 모델**을 써라.)

(생성보다 검증이 쉽다는 "NP 스타일" 직관은 여기서 오용되기 쉽다. NP는 증명서가 주어진 경우의 성질이다.)

**진단 지표:** 체커 반려율. 너무 낮으면 체커가 놀고 있고(담합), 너무 높으면 maker 스펙이 나쁘다. 절대 수치보다 **의도적 결함 diff를 주기적으로 주입해 탐지율을 재는 것**이 신뢰도가 높다.

## 4. 게이트: 순서가 전부다

정렬 기준은 "싼 것부터"가 아니라 **결정론성 우선, 그다음 비용**이다.

```
0. 린트 / 타입체크 / 포맷       결정론적, ~1초   → 실패 시 워커에 즉시 반송 (사람 X)
1. 정책 게이트                  결정론적, ~1초   → 시크릿? 마이그레이션? 테스트·CI 파일 수정?
                                                   테스트 실행 전에 건다 — 변조된 스위트를 돌리지 않기 위해
2. 유닛 테스트 (maker 가시)     결정론적, ~30초  → 실패 시 워커에 즉시 반송 (사람 X)
3. 통합 / e2e                   결정론적
4. Eval 스위트 (골든셋)         준결정론적       → 회귀 시 배포 차단
5. LLM-as-judge (스펙 준수)     비결정론적       → 가장 마지막, 가장 못 믿음
─────────────────────────────────────────────
6. 홀드아웃 스위트 (maker 비가시) ← 별도 축. 아래 참조
```

**린트·유닛(0·2단계) 실패는 사람에게 알리지 않는다.** 에러를 워커에 그대로 던지고 다시 돌린다. 이게 루프다. 단, 무통보 재시도에도 **반복 실패 상한**은 걸어야 한다(→ 서킷브레이커). **정책 게이트(1단계)는 예외** — 무통보 재시도 대상이 아니다: 테스트·CI 변조는 하드블록/에스컬레이트(§7과 일치), 시크릿·마이그레이션은 워커 반송이나 사람 게이트로 분기. 사람에게 올라오는 나머지 신호는 eval(4단계) 이상만. 그렇지 않으면 알림 피로로 인간 게이트가 무의미해진다.

**"녹색"의 정의** = 에이전트 자기보고가 아니라 **사람이 겪는 그 CI가 통과**한 것. 비가역 명령군(merge/deploy/publish/삭제)은 프롬프트 지침이 아니라 **훅/정책으로 기계적 차단**.

### 홀드아웃은 루프 밖에 둔다 ★

**SpecBench** (Weco AI, arXiv:2605.21384) 결과:

- 모든 프론티어 에이전트가 **가시 테스트 스위트를 포화**시킨다
- 그럼에도 홀드아웃 격차는 사라지지 않고, **소형 모델일수록 격차가 크다**
- 격차는 **코드 규모 10배당 28%p씩 확대**된다
- 극단 사례: 테스트 입력을 암기하는 2,900줄짜리 해시테이블 "컴파일러"

즉 리워드 해킹은 엣지 케이스가 아니라 **태스크 길이에 따라 구조적으로 증가하는 함수**다.

> **홀드아웃 누출 방지 규칙:** 홀드아웃 실패 상세를 워커에 되먹이면 몇 번의 반복 후 홀드아웃은 사실상 가시 테스트가 된다. 홀드아웃은 **최종 게이트에서 pass/fail 비트만** 반환하고, 실패 상세는 사람에게만 간다. 같은 홀드아웃을 두드리는 횟수도 예산화한다.

### Eval 게이트

도구(Braintrust / LangSmith / Promptfoo / Arize Phoenix / DeepEval)는 중요하지 않다. 중요한 건 둘:

- **골든 데이터셋이 프로덕션 실패에서 계속 자라는가.** 수백 개로 시작하지 말고 실제 실패에서 뽑은 20~50개로 시작한다.
- **임계값이 회귀 델타 + 절대 하한 둘 다인가.** 절대값만 걸면 팀이 임계값을 낮춰 무력화한다. 회귀 델타만 걸면 매 릴리스 -1%씩 영원히 통과하는 **완만한 드리프트**를 놓친다.

LLM-judge는 반드시 **사람 라벨로 캘리브레이션**한다. LLM-facing 동작은 단위 테스트로 안 잡힌다 — eval 스위트 + 회귀 델타가 유일한 신호다.

## 5. 사람은 비가역 경계에만

**원칙:** 사람은 "확인"하지 않는다. 사람은 **되돌릴 수 없는 것만 승인한다.**

행동을 **가역성 × 폭발반경**으로 분류한다. 질문을 "물어봐야 하나?"가 아니라 **"되돌릴 수 있나?"**로 바꾸는 게 설계의 축이다.

| 티어 | 정의 | 예시 | 정책 |
|---|---|---|---|
| **T0** | 가역 · 로컬 | 편집, 테스트, 로컬 커밋, 작업브랜치 push, draft PR | 자율 — 재확인 금지 |
| **T1** | 가역 · 공유/관측 | 스테이징 배포, 이슈 코멘트, 내부 대시보드 갱신, 캐시 무효화 | 실행하되 기록 |
| **T2** | 비가역 · 고임팩트 | main 머지, 프로덕션 배포, publish, 외부 발신, 데이터 삭제, 자금 이동, 시크릿·IAM 변경, PII 반출 | 사람 게이트 |

**비가역 판정 기준 (하나라도 걸리면 T2):**
- 되돌리는 비용이 만드는 비용보다 크다
- 되돌림이 관측 불가능하다
- 외부인이 이미 봤다

이 기준을 적용하면 **대부분의 코드 변경은 가역**이다. 브랜치 커밋, 스테이징 배포, 피처 플래그 뒤의 프로덕션 배포까지 전부. 그래서 사람 승인 대상이 아니다. 사람은 나중에 읽는다(human-on-the-loop). 표준 형태는 **PR 리뷰 → 머지**이고, 그 앞은 전부 자율이다. 사람 주의력은 배치로 소비한다 — 건마다 인터럽트하지 말고 PR + 요약 하나를 async로.

### 가장 중요한 설계 이동 ★

승인 요청을 줄이는 최선의 방법은 승인을 잘 처리하는 게 아니라 **비가역 행동을 가역으로 바꾸는 것**이다.

| 비가역 | → 가역 변환 |
|---|---|
| 이메일 발송 | 발송 큐 + 15분 지연 (그 사이 취소 가능) |
| 스키마 마이그레이션 | expand/contract 패턴 (drop을 다음 릴리스로 분리) |
| 프로덕션 배포 | 피처 플래그 + 카나리 (즉시 롤백) |
| 파일 삭제 | tombstone + 30일 GC |

> **자율성은 모델을 믿어서 얻는 게 아니라 되돌림 가능성을 엔지니어링해서 얻는다.**

**승인 UI 주의:** diff 500줄을 던지면 사람은 승인을 누른다. 무조건. 인간 게이트가 실제 게이트이려면 요청은 (a) 무엇이 비가역인지 한 줄, (b) 되돌림 계획, (c) 반대 근거 — 셋으로 압축돼야 한다. 압축을 못 하겠으면 그건 승인할 준비가 안 된 변경이다.

## 6. 관측 · 롤백 · Kill-switch

### 관측

**단위는 "요청"이 아니라 "트레이스"다.** 에이전트 한 실행 = 하나의 트리(오케스트레이터 스팬 → 워커 스팬 → 툴콜 스팬). OpenTelemetry GenAI 시맨틱 컨벤션에 맞추면 기존 APM에 얹힌다. 무인 플릿은 조용히 실패하므로 가시성이 디버깅의 전부다.

| 메트릭 | 왜 |
|---|---|
| 턴당 토큰·비용 | 폭주 감지 (선행지표) |
| 동일 툴콜 반복 횟수 | 루프 고착의 가장 빠른 신호 |
| 체커 반려율 | maker 품질 / 스펙 품질의 프록시 |
| **가시 vs 홀드아웃 통과율 격차** | **리워드 해킹의 직접 지표** |
| 사람 승인 요청 건수 | 자율성 회귀의 지표 |
| 태스크당 비용 · 에스컬레이션율 | 경제성 |

### 서킷 브레이커 (자동, 인프라 계층)

5분 내 툴콜 N회 초과 / 동일 파라미터 툴콜 3회 연속 / 세션 비용 상한 초과 → 일시정지 + 알림.

**모델의 판단을 신뢰하지 않고 프로세스 밖에서 센다.** 에이전트가 스스로 멈추길 기대하는 설계는 실패한다. 하드캡(iteration · token · 시간 · 일일 비용)은 협상 대상이 아니다.

### Kill-switch의 냉정한 진실

에이전트를 죽여도 **이미 한 일은 되돌아가지 않는다.** kill-switch는 미래를 막을 뿐 과거를 지우지 않는다. 그래서 kill-switch는 **롤백 계획과 짝으로만** 의미가 있고, 롤백 계획은 사고 전에 정의돼야 한다.

최소 요건:

- **단일 스톱** — 사람, 자동 모니터, 최종 사용자 누구든 이유 설명 없이 누를 수 있어야 한다
- **상태 스냅샷** — 스톱 시점의 트레이스 + 워크트리 + 열린 외부 핸들
- **부수효과 원장(side-effect ledger)** — 에이전트가 수행한 **비가역 행동의 append-only 로그**. 롤백은 이 원장을 역순으로 읽으며 보상 트랜잭션을 실행한다. saga 패턴 그대로다.

**kill-switch 권한 자체는 에이전트가 만질 수 없는 곳에 둔다.** 에이전트가 자기 정지 정책이 있는 리포에 쓰기 권한을 갖고 있으면 그건 kill-switch가 아니다. 샌드박스 · 최소권한 · egress 통제로 폭발반경을 격리한다.

## 7. 실패 모드 → 대응

| 실패 모드 | 증상 | 대응 |
|---|---|---|
| **리워드 해킹** | 가시 테스트 100%, 홀드아웃 붕괴. 하드코딩·테스트 조작 | 홀드아웃 스위트(루프 밖), 테스트 파일 쓰기 권한 분리, 런 중 수용기준 동결, 코드 규모 이상 증가 알람 |
| **테스트 하네스 변조** | `pytest.ini`, CI 설정, 테스트 자체를 수정 | 정책 게이트에서 diff 경로 차단. 테스트·CI 파일 변경은 **항상 비가역 취급** |
| **Silent green** | rubric 통과인데 실제론 깨짐 | 독립 검증 다층화, 실동작 e2e, 캘리브레이션된 judge, 사람 스팟체크 |
| **컨텍스트 부패** | 장기 실행 후 계획 능력 저하, 초반 지시 망각 | 오케스트레이터 컨텍스트 격리, 태스크마다 fresh 컨텍스트를 durable state에서 재구성, 짧은 워커 |
| **루프 고착 / 비용 폭주** | 같은 툴콜 반복, 무한 재시도 | 하드캡 + 서킷브레이커. 3회 실패 시 다른 워커 + 다른 모델로 재발행 |
| **체커 담합** | 반려율 극저, 통과한 코드가 프로덕션에서 실패 | 정보/권한 비대칭 강제. 의도적 결함 diff 주기적 주입으로 탐지율 측정 |
| **팬아웃 발산** | 워커들이 상충하는 전제로 작업, 머지 불가 | 오케스트레이터가 인터페이스 계약을 먼저 고정, 그다음 팬아웃 |
| **오류 누적** | 작은 오류가 복리로 증폭 | 자주 체크포인트 + 검증, 작은 가역 스텝, 롤백 |
| **과잉 확인** | 사소한 가역 행동을 물어봄 → 병목 | 가역성 분류로 T0 재질문 금지 |
| **과소 확인** | 게이트 없이 비가역 실행 | 명령군 기계적 하드블록(훅/정책) |
| **버전 드리프트** | 모델 업그레이드 후 조용한 행동 변화 | 모델 버전 핀 + 업그레이드를 eval 게이트 통과 대상으로 취급(코드 변경과 동일) |
| **승인 피로** | 사람이 전부 승인 누름 | 요청 건수를 SLO로 관리. 초과 시 **자율성 설계 결함**으로 간주 |
| **Flakiness** | 재현 안 되는 실패 | 멱등 재시도, flaky 격리, seed 고정 |
| **공유 워킹트리 엉킴** | 한 워킹트리를 두 세션이 공유 → auto-commit(`git add -A`)이 상대 세션의 미커밋 변경까지 한 커밋에 스테이징, 히스토리 오염 | 스테이징을 **세션 단위로 스코프**: PostToolUse가 세션별 `.touched-<sid>` 매니페스트를 기록하고, 다른 살아있는 세션이 감지되면(`loop_lock.sh others`) auto-commit이 `add -A` 대신 내 매니페스트 경로만 스테이징. 혼자면 기존 `add -A` 백스톱. auto_pr만 동일-세션 `.run-marker` 게이트 유지 + 워킹트리당 단일-루프 락(`loop_lock.sh`, 두 번째 loop-run 거부) + 병렬 필요 시 worktree 격리(§2) |

## 8. 트레이드오프 (솔직하게)

- **자율 ↔ 검증**: 자율을 올리면 하류의 검증·리뷰로 지불한다. 공짜 자율은 없다.
- **팬아웃 폭 ↔ 비용**: 병렬 워커는 정확도를 사지만 토큰이 뛴다. 태스크 가치가 토큰 비용을 넘을 때만 정당화된다. 비용 곡선은 선형이 아니다.
- **멀티 ↔ 단일**: 스웜은 조율 오버헤드·비예측 → 제약 그래프 선호. 병렬/독립 관점이 **실제로 필요할 때만** 에이전트를 늘린다.
- **게이트 엄격도 ↔ 처리량**: 게이트를 조이면 반려율이 올라가고 워커 재시도로 비용이 뛴다. 풀면 프로덕션 사고 비용을 낸다. **게이트 비용 < 사고 기대비용**이 되는 지점을 실제로 계산해라. 대부분 팀은 직관으로 정한다.
- **오케스트레이터 지능 ↔ 지연**: 똑똑한 오케스트레이터는 좋은 분해를 하지만 계획에만 수 분을 쓴다. 태스크가 짧으면 계획이 실행보다 비싸다.
- **코드 ↔ 모델**: 예산·게이트 판정·상태 파싱 같은 결정적 로직은 스크립트로. 모델은 판단에만.
- **자율성 ↔ 감사가능성**: 사람이 덜 볼수록 트레이스는 더 좋아야 한다. 관측 투자 없이 자율성만 올리면 사고 시 재구성이 불가능하다.

## 9. 시공 순서

이 순서를 어기면 대부분 실패한다:

1. **먼저 검증기.** 홀드아웃 스위트와 eval 골든셋 없이는 루프를 켜지 마라.
2. **그다음 관측.** 트레이스와 카운터. 볼 수 없으면 자율성을 늘릴 근거가 없다.
3. **그다음 kill-switch와 부수효과 원장.**
4. **그다음 단일 워커 루프.** 자동 재시도만.
5. **그다음 체커 분리.**
6. **마지막에 팬아웃.**

팬아웃부터 짓는 팀이 많다. 화려하고 데모가 잘 나오기 때문이다. 그 팀들은 6개월 뒤 **"에이전트가 만든 코드를 아무도 못 믿는"** 상태에 도달한다. 검증기가 없어서다.

## 10. Claude Code(loopy)에서의 구현

§1~9는 도구 비종속 독트린이다. 이 절은 그 독트린이 **이 레포(loopy 플러그인)에서** 어떤 Claude Code 프리미티브로 구현되는지의 매핑이다 — 진단·리뷰의 **기준이 아니라 참조 구현**이다(요구사항은 §1~9와 ETCLOVG에 있고, `loop-architect`·`design-critic`는 그쪽을 채점한다). 상태: ✅ 구현 · ⚠ 부분 · ✕ 독트린만(미탑재).

| 개념 (§) | loopy 프리미티브 | 상태 |

| 오케스트레이터 · durable state (§1·2) | 메인 에이전트(세션) + `.claude/loop/` (`goal`·`rubric`·`state`·`memory`·`review`·`loop.config`.md) | ✅ |

| 예산 (§8) | `scripts/check_budget.sh`(description 합계 ≤300단어 · SKILL 본문 ≤500단어) + `codex_args` | ✅ |
| 시공 순서 (§9) | `loop-architect` 성숙도 사다리 L0~L5로 기계화 | ✅ |

운영 절차(사이클 형태·게이트 배선·메모리 프로토콜)는 여기서 반복하지 않는다 — Claude Code 특화 운영 카운터파트는 `skills/loop-engineering/SKILL.md`와 그 `references/`다.

✕/⚠ 행(홀드아웃·eval 게이트·테스트/CI 변조 차단·부수효과 원장·독립 kill-switch·관측 카운터·가역성 엔지니어링)이 곧 `loop-architect`가 이 레포에 매기는 **성숙도 상한(L4/L5)의 사유**다. 진단·리뷰 기능이 존재하는 이유가 바로 이 갭을 기계적으로 드러내기 위해서다 — 문서가 스스로를 채점한다.
